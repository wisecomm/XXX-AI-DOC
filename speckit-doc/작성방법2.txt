ㄱ-작성방법2

> speckit.specify 작성해야 할 내용 설명해줘

✦ speckit.specify 파일은 AI 소프트웨어 엔지니어 에이전트에게 특정 작업이나 기능 구현에 대한 상세한 요구사항과 명세를 전달하기 위해 사용되는 파일입니다.

  speckit.constitution이 AI가 프로젝트 전반에 걸쳐 지켜야 할 일반적인 "헌법"이라면, speckit.specify는 AI가 수행해야 할 개별 작업에 대한 구체적인 "설계도" 
  또는 "작업 지시서"라고 할 수 있습니다.

  이 파일은 AI가 작업을 정확하게 이해하고, 요구사항에 맞춰 구현하며, 올바르게 검증할 수 있도록 돕는 역할을 합니다.

  일반적으로 speckit.specify 파일에 포함될 수 있는 내용은 다음과 같습니다.

  주요 내용

   1. 작업 설명 (Task Description)
       * AI가 수행해야 할 작업의 명확하고 간결한 설명 (예: "사용자 로그인 기능 구현", "장바구니 버그 수정", "데이터베이스 스키마 변경").
       * 작업의 목표와 기대 결과.

   2. 범위 (Scope)
       * 이 작업이 영향을 미치는 코드베이스의 영역.
       * 작업에 포함되는 내용과 제외되는 내용을 명확히 정의.

   3. 기능 요구사항 (Functional Requirements)
       * 구현될 기능이 '무엇을 해야 하는지'에 대한 구체적인 설명.
       * 사용자 인터페이스(UI) 요소, 데이터 처리 로직, 시스템 동작 방식 등.

   4. 비기능 요구사항 (Non-Functional Requirements)
       * 성능 (예: 응답 시간, 처리량), 보안 (예: 데이터 암호화), 사용성, 확장성, 유지보수성 등 기능 외적인 요구사항.

   5. 수락 기준 (Acceptance Criteria)
       * 작업이 성공적으로 완료되었음을 판단하는 명확하고 측정 가능한 조건.
       * 테스트 케이스 형태로 작성될 수 있으며, AI가 자체적으로 검증할 수 있는 기준을 제공합니다. (예: "유효한 자격 증명으로 로그인 시 성공", "잘못된 비밀번호
          입력 시 에러 메시지 표시").

   6. 기술 스택 및 제약사항 (Technology Stack & Constraints)
       * 이 작업을 구현하는 데 사용해야 할 특정 기술, 라이브러리, 프레임워크.
       * 특정 디자인 패턴, 아키텍처 원칙 또는 성능 제약사항.

   7. 참조 자료 (References)
       * 관련 문서, 디자인 시안, API 명세, 기존 코드 예시 등 AI가 작업을 수행하는 데 도움이 될 수 있는 자료 링크.

   8. 테스트 지침 (Testing Instructions)
       * AI가 구현된 기능을 어떻게 테스트해야 하는지에 대한 지침. (예: "새로운 기능에 대한 단위 테스트를 작성하고, 기존 통합 테스트를 실행하여 회귀 여부를
         확인").

  간단한 예시

    1 # speckit.specify 예시 파일: 사용자 로그인 기능 구현
    2 
    3 # 1. 작업 설명
    4 - 사용자가 이메일과 비밀번호를 사용하여 시스템에 로그인할 수 있는 기능을 구현합니다.
    5 - 성공적인 로그인 시 사용자 대시보드로 리디렉션됩니다.
    6 - 로그인 실패 시 적절한 오류 메시지를 표시합니다.
    7 
    8 # 2. 범위
    9 - 프론트엔드: 로그인 폼 UI (React 컴포넌트)
   10 - 백엔드: 로그인 API 엔드포인트 (/api/auth/login)
   11 - 데이터베이스: 사용자 인증 정보 조회 로직
   12 
   13 # 3. 기능 요구사항
   14 - 사용자는 이메일 주소와 비밀번호를 입력할 수 있어야 합니다.
   15 - "로그인" 버튼을 클릭하면 인증 요청이 백엔드로 전송됩니다.
   16 - 백엔드는 이메일과 비밀번호를 검증하고, 유효한 경우 JWT 토큰을 반환합니다.
   17 - 프론트엔드는 JWT 토큰을 로컬 스토리지에 저장합니다.
   18 - 로그인 성공 시 `/dashboard` 경로로 이동합니다.
   19 - 이메일 형식이 유효하지 않거나, 비밀번호가 6자 미만일 경우 클라이언트 측 유효성 검사를 수행합니다.
   20 
   21 # 4. 비기능 요구사항
   22 - 로그인 API 응답 시간은 500ms 이내여야 합니다.
   23 - 비밀번호는 해싱되어 저장되어야 하며, 전송 시 HTTPS를 사용해야 합니다.
   24 
   25 # 5. 수락 기준
   26 - [ ] 유효한 이메일과 비밀번호로 로그인 시 성공하고 대시보드로 이동한다.
   27 - [ ] 잘못된 비밀번호 입력 시 "비밀번호가 일치하지 않습니다." 메시지가 표시된다.
   28 - [ ] 존재하지 않는 이메일로 로그인 시 "등록되지 않은 사용자입니다." 메시지가 표시된다.
   29 - [ ] 이메일 입력 필드가 비어있을 경우 "이메일을 입력해주세요." 메시지가 표시된다.
   30 - [ ] 비밀번호 입력 필드가 비어있을 경우 "비밀번호를 입력해주세요." 메시지가 표시된다.
   31 
   32 # 6. 기술 스택 및 제약사항
   33 - 프론트엔드: React, TypeScript, Material-UI, React Hook Form
   34 - 백엔드: Node.js, Express, Passport.js, JWT
   35 - 데이터베이스: PostgreSQL
   36 - 기존 `User` 모델을 활용합니다.
   37 
   38 # 7. 참조 자료
   39 - 디자인 시안: [링크]
   40 - API 명세: [링크]

  speckit.specify 파일을 통해 AI는 주어진 작업을 명확하게 이해하고, 개발 과정에서 발생할 수 있는 불확실성을 줄여 더 정확하고 효율적으로 작업을 수행할 수
  있게 됩니다.
